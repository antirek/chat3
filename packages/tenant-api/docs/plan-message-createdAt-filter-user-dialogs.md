# План: фильтр `message.createdAt` в `GET /api/users/:userId/dialogs`

## 1. Цель

Поддержать семантику: **вернуть только те диалоги пользователя, в которых за указанные условия по времени существует хотя бы одно сообщение** (по полю `Message.createdAt`).

Это **не** то же самое, что `lastMessageAt` в `UserDialogActivity` (время последнего сообщения).

---

## 2. Публичный контракт (что обещаем клиентам)

### 2.1. Формат в параметре `filter`

- **Строковый синтаксис** (как остальные фильтры tenant-api), с вложенным полем через точку:
  - `(message.createdAt,gte,<timestamp>)`
  - `(message.createdAt,lte,<timestamp>)`
  - комбинация через `&` для закрытого интервала:  
    `(message.createdAt,gte,T1)&(message.createdAt,lte,T2)`

Значения времени в **операндах фильтра** — **unix-time**; на практике клиенты часто передают **целые секунды** без дробной части. В **БД** у `Message.createdAt` — число в той же логике, что и `generateTimestamp()` (миллисекунды unix + дробная часть). При реализации нужно **явно согласовать шкалу** (например интерпретировать операнд в секундах как `* 1000` для сравнения с полем, либо документировать «только миллисекунды») — зафиксировать в коде и в ответе API, чтобы не было молчаливого сдвига на три порядка.

Сценарий «вчера» не специализируем — важен случай **«сообщения в интервале»** (границы задаёт клиент).

Операторы: минимум **`gte`**, **`lte`**, **`gt`**, **`lt`**; при необходимости **`eq`**. Допускается **только одна граница** (например только `gte` или только `lte`) — полуоткрытый интервал; поведение зафиксировать в тестах.

**Окно по времени (продукт):** если после нормализации условия по `createdAt` заданы **две конечные границы** (есть и нижняя, и верхняя — не важно, какими операторами), то **разница между ними не больше 24 часов** в **одной согласованной шкале** (после приведения секунд/мс к одному виду). Если граница **одна** — ограничение «24 часа» на длину интервала **не применяем** в v1 (иначе противоречило бы полуинтервалам). Типичный закрытый интервал от клиента в **секундах** (`gte` начала суток + `lte` конца суток) — это **не** «нулевой интервал», а окно до ~86400 секунд; проверка 24ч должна использовать **те же единицы**, что и вычисленная длина окна.

**`message.type` и прочие поля сообщения:** фильтр **`message.type`** и любые другие поля под **`message.*`, кроме `createdAt`, в этой итерации не поддерживаем** — отдельная задача. Если в фильтре присутствует **`message` с ключами, отличными от `createdAt`,** — **400 Validation error** (формулировка: неподдерживаемые поля в блоке `message`).

### 2.2. Сочетание с другими фильтрами

Логика **И (AND)** с уже существующими ограничениями по `dialogIds` (meta, topic, member и т.д.): сначала или потом вычислять множество `dialogId` из сообщений — порядок на уровне реализации, результат — пересечение множеств.

### 2.3. Логическое ИЛИ в строке фильтра (`|`) и условие по сообщениям

**О чём речь (переформулировка «вопроса про $or»):** в синтаксисе фильтра есть **два уровня логики** — между атомами через **`&`** (И) и через **`|`** на верхнем уровне группы (**ИЛИ**). После парсинга цепочка с `|` превращается в MongoDB-объект с **`$or`**. Вопрос был: *если клиент напишет, например, `(meta.type,eq,internal)|(message.createdAt,gte,T)` — обязаны ли мы отбирать диалоги, где выполняется одно **или** другое?* Это усложняет реализацию (разные ветки — разные способы получить множество `dialogId`).

**Решение для v1:** при наличии в запросе **`message.createdAt`** (или любого будущего поддерживаемого поля под `message.*`) **не поддерживаем комбинацию с верхнеуровневым ИЛИ** в том же `filter`. Практически: если распарсенный фильтр содержит **`$or`** (из-за `|` в строке) **и** при этом участвует блок **`message`** — отвечать **`400 Bad Request`** с понятным текстом (например: «условия по сообщениям несовместимы с оператором ИЛИ в фильтре; используйте только `&`»). Фильтры **только с `&`**, включая `(message.createdAt,...)&(meta....)&...`, **разрешены**.

### 2.4. Прочие ограничения v1

- Фильтр применяется к сообщениям тенанта **`tenantId` из запроса**; дополнительных правил по видимости сообщений для пользователя в v1 не вводим (участие пользователя в диалоге уже задаётся через `DialogMember`).
- Валидация **максимальной длины окна 24ч** — при реализации: вычислить нижнюю и верхнюю границу из нормализованного условия по `createdAt`; если обе заданы и **верхняя − нижняя > 24h** — **400**.

## 3. Backend (tenant-api)

### 3.1. Основная идея

1. После `parseFilters` / `extractMetaFilters`, если в `regularFilters` есть **`message`** с полем **`createdAt`**, **не** передавать этот объект в `dialogMembersQuery`.
2. Нормализовать условие по `createdAt` в один MongoDB-фильтр для поля `createdAt` (объединить `$gte`/`$lte`/`$gt`/`$lt`/`$eq` в один объект, без дублирования ключей на `Message`). Перед запросом: проверить **окно 24ч** при двух границах; отклонить лишние ключи в **`message`** (не `createdAt`).
3. Выполнить запрос к коллекции сообщений, например:
   - `Message.distinct('dialogId', { tenantId, createdAt: condition })`  
   или агрегацию `$match` + `$group`, если понадобится дедупликация иначе.
4. Пересечь полученный список `dialogId` с множеством диалогов пользователя:
   - либо **фильтр в памяти** после получения `distinct` и списка `dialogId` пользователя,
   - либо **`dialogId: { $in: [...] }`** в запросе к `Message`, предварительно загрузив `dialogId` участника из `DialogMember` (для больших тенантов может быть тяжелее; для v1 часто достаточно `distinct` + пересечение с небольшим/средним объёмом).
5. Результат пересечения влить в существующую переменную **`dialogIds`** (как для meta-фильтров): если `dialogIds === null`, присвоить; иначе — **пересечение**. Если пересечение пустое — вернуть пустой список с корректной пагинацией.
6. Удалить из `regularFilters` ключ **`message`**, чтобы вложенные поля не попали в неверный запрос к `DialogMember` (аналогично обработке `topic`).

### 3.3. Индексы

В `packages-shared/models/src/data/Message.ts` уже есть индекс `{ tenantId: 1, createdAt: -1 }` — для запросов по диапазону `createdAt` с `tenantId` этого достаточно для старта. При медленных профилях в проде — пересмотреть explain-планы.

### 3.4. Производительность и пагинация

- Ограничение **«не больше 24 часов между границами»** при двух концах снижает риск полного скана огромного исторического диапазона; полуоткрытые интервалы остаются без этого ограничения — риск нагрузки учитывать в мониторинге.
- **Итоговая пагинация** должна считаться **после** всех фильтров (как сейчас для `getUserDialogs`), чтобы `total`/`pages` были согласованы.

### 4.2. Тесты на поведение «вопрос 1»: ИЛИ (`|`) несовместимо с `message.*`

Цель — зафиксировать решение §**2.3**: при **верхнеуровневом** `$or` в распарсенном фильтре и наличии блока **`message`** — ответ **`400`**, без выполнения выборки по сообщениям.

**Замечание для реализации:** детектировать связку **`$or` + `message`** по объекту после `parseFilters` (рекурсивный обход ключей `message` и `$or`), до тяжёлой логики meta/topic.

## 7. Риски

- **Нагрузка:** окно **24ч при двух границах** ограничивает скан; при **одной** границе (разрешено продуктом) возможен тяжёлый запрос — мониторинг и при необходимости доработка в отдельной задаче.
- **Дублирующие условия** на `message.createdAt` через `&` и верхнеуровневый `$and` в `regularFilters` — не должны попадать в `DialogMember`; при извлечении блока `message` сливать границы в одно условие по `createdAt` до запроса к `Message`.
- **Проверка `$or` + `message`:** реализовать рано в обработке `filter` (по распарсенному дереву), чтобы не смешивать неподдерживаемую семантику.
- Согласованность с **AGENTS.md**: фильтр по **`message.type`** — отдельная задача; пространства имён сообщений при расширении — проверять отдельно.

## 8. Контрольный анализ плана (актуальность и пробелы)

### 8.1. Согласованность

- **§2.1–2.4**, **§3.1** и таблица **§4.1** согласованы: границы, 24ч, только `message.createdAt`, запрет лишних ключей.
- **§2.3** явно закрыт тестами **§4.2** (OR-1…OR-5), в **§3.2** добавлена ранняя проверка — план реализуем без противоречий.
- **§7** и ограничение 24ч совпадают с валидацией в **§3.1** / строкой **#10** в тестах.

### 8.2. Что всё ещё предстоит уточнить в коде (не блокеры плана)

- **Вычисление «двух границ»** для правила 24ч: при комбинации `gt`+`lt` без равенств границы всё равно две — формулы min/max по веткам условий нужно зафиксировать в одном месте (утилита `getCreatedAtBounds(condition)`).
- **`eq` и правило 24ч:** при **unix-секундах** у клиента **`eq`** по смыслу ближе к **интервалу длительностью 1 секунда** (гранулярность операнда), а не к математической точке нулевой длины и не к «двум одинаковым границам». Проверку «две границы и ≤ 24ч» применять к парам **`gte`+`lte`**, **`gt`+`lt`** и т.п.; для **`eq`** — отдельный тест с ожидаемой семантикой попадания сообщений в пределах **этой 1 с** после согласования шкалы с БД.

### 8.3. Готовность к реализации

Плана **достаточно**, чтобы начать код: контракт, порядок шагов, индекс, таблица тестов и отдельный блок по **ИЛИ + message**. Оставшиеся пункты **§8.2** — утилита границ с **единой шкалой** (секунды клиента ↔ хранилище) и тест на **`eq`** в модели **интервала ~1 с** (гранулярность секунд), а не нулевой длины.
